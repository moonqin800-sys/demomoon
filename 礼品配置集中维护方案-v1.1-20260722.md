# 礼品配置集中维护方案

## 1. PPT 内容分析结论

本次 PPT 共 5 页，核心表达的是：现有礼品相关配置分散在多个后台入口，导致同一个礼品的图片、名称、详情、活动展示信息需要在不同地方重复维护，前端消费者页面又依赖这些配置进行展示。

### 1.1 当前涉及的后台与前端位置

| 场景 | PPT 页码 | 当前位置 | 看到的能力/字段 | 前端影响 |
|---|---:|---|---|---|
| 礼品基础展示 | 第 1 页 | FMC 端礼品区域 | 上传物料图、产品名称、赠品用图配置 | 消费者端“我的礼品-实物礼品”展示礼品缩略图和标题 |
| 活动页礼品展示 | 第 2 页 | 活动管理 > 礼品管理 | 产品编码、产品名称、产品类型、品牌、产品图片、物料图片、活动用图 | 消费者端活动页“请选择赠品”卡片展示 |
| 礼品详情展示 | 第 3 页 | FMC 端礼品详情 | 礼品详情图配置、服务电话等详情信息 | 消费者点击“礼品详情”后的弹层/详情页 |
| 活动绑定礼品 | 第 4 页 | 礼品展示管理 > 礼品展示列表 > 配置礼品 | 礼品代码、礼品名称、客服电话、图片地址、视频地址、排序、状态 | 活动内展示哪些礼品、排序、启停 |
| 同步礼品展示 | 第 5 页 | 礼品展示管理 > 添加礼品展示 | 礼品名称、礼品 code、客服电话、礼品别名、礼品封面图、礼品视频 | 礼品展示列表及详情展示 |

### 1.2 当前主要问题

1. 礼品信息分散：礼品管理、礼品展示管理、活动配置里都有礼品字段，维护入口不统一。
2. 图片口径不清：产品图片、物料图片、活动用图、礼品详情图、图片地址混用，业务人员难判断该传哪张图。
3. 重复录入风险高：礼品名称、code、客服电话、图片、视频等字段在不同页面重复出现，容易出现前后端展示不一致。
4. 活动配置和礼品主数据耦合：活动里既在“选礼品”，又在“编辑礼品内容”，导致一个活动改图是否影响其他活动不清晰。
5. 前端消费链路缺少统一数据源：消费者端活动页、我的礼品、礼品详情可能读取不同配置表，导致展示不一致。

## 2. 产品整合目标

建议将后台收敛为“礼品中心”统一维护礼品主数据，并把活动侧能力简化为“选择礼品 + 活动级配置”。

核心原则：

1. 一个礼品只维护一份主资料。
2. 礼品图片按使用场景明确命名和预览。
3. 活动只负责绑定、排序、启停和必要的活动级覆盖。
4. 消费者端所有礼品展示统一读取礼品中心数据，活动页再叠加活动绑定关系。

## 3. 建议后台架构

### 3.1 菜单整合

建议保留或升级现有“礼品管理”为统一入口，命名为：

**礼品中心**

下设 3 个页面：

| 页面 | 作用 | 替代/吸收现有能力 |
|---|---|---|
| 礼品库 | 承接上游同步的礼品主数据，补充图片、详情、视频、客服电话 | 吸收“礼品管理”“添加礼品展示”里的重复字段 |
| 礼品素材 | 按场景维护图片/视频资源，可作为礼品库编辑页内 Tab | 吸收“赠品用图配置”“活动用图”“礼品详情图配置” |
| 活动礼品配置 | 按活动选择礼品、设置排序、启停、库存展示策略 | 吸收“礼品展示列表 > 配置礼品” |

不建议把所有字段硬塞进同一张表或同一个巨大表单。更合理的是：一个后台入口，两个数据层级。

### 3.2 数据层级

| 层级 | 维护内容 | 是否全局生效 |
|---|---|---|
| 礼品主数据 | 由上游系统同步礼品 code、名称、品牌、类型、规格等基础资料；后台补充客服电话、主图、活动图、详情图、视频、状态 | 是 |
| 活动礼品关系 | 活动 code、礼品 code、排序、活动内启停、展示时间、活动内别名/文案覆盖、库存或限量策略 | 只对当前活动生效 |

这样可以满足“后台配置放到一个地方维护”，同时避免某个活动临时换图影响历史活动或其他活动。

## 4. 字段设计建议

### 4.1 礼品库字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| 礼品 code | 文本 | 是 | 全局唯一，作为各活动绑定主键 |
| 礼品名称 | 文本 | 是 | 后台与前端默认展示名称 |
| 礼品别名 | 文本 | 否 | 可用于短标题或消费者端展示 |
| 礼品类型 | 枚举 | 是 | 实物礼品、红包礼品、虚拟礼品等 |
| 品牌 | 枚举/文本 | 否 | 如 Friso、Hape 等 |
| 规格/型号 | 文本 | 否 | 如 3 段 800g、毛巾浴巾等 |
| 客服电话 | 文本 | 否 | 礼品详情页展示 |
| 礼品状态 | 枚举 | 是 | 草稿、启用、停用 |
| 主图 | 图片 | 是 | 用于礼品库列表、我的礼品等通用缩略图 |
| 活动展示图 | 图片 | 是 | 用于消费者活动页“请选择赠品”卡片 |
| 详情图 | 图片 | 否 | 用于礼品详情弹窗/详情页 |
| 物料图 | 图片 | 否 | 用于线下物料、登记入库或运营下载 |
| 视频地址 | URL/上传 | 否 | 用于礼品详情页视频展示 |
| 备注 | 多行文本 | 否 | 内部运营说明 |

图片建议在上传区域直接标注使用位置，例如“活动展示图：消费者活动页礼品卡片”。不要只叫“上传图片”，否则运营仍然会传错。

### 4.2 活动礼品配置字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| 活动 code | 文本/选择 | 是 | 当前活动 |
| 礼品 code | 选择 | 是 | 从礼品库选择，不手工录入 |
| 排序 | 数字 | 是 | 活动页展示顺序 |
| 活动内状态 | 枚举 | 是 | 启用、停用 |
| 展示开始/结束时间 | 时间 | 否 | 不填默认跟随活动 |
| 活动内展示名称 | 文本 | 否 | 不填使用礼品库名称 |
| 活动内展示图 | 图片 | 否 | 不填使用礼品库活动展示图 |
| 库存/剩余量展示 | 开关/配置 | 否 | 用于“当前剩余 X 个礼品”等文案 |

活动侧允许覆盖字段，但必须清晰提示：不填写则继承礼品库；填写后仅当前活动生效。

## 5. 后台交互流程

### 5.1 同步礼品流程

1. 运营进入“礼品中心 > 礼品库”。
2. 点击“同步礼品”。
3. 系统从上游礼品主数据系统拉取礼品 code、名称、类型、品牌、规格等基础信息。
4. 运营补充客服电话、主图、活动展示图、详情图、物料图。
5. 系统校验同步结果、图片尺寸、格式、大小。
6. 保存展示素材配置并启用。
7. 启用后可被活动配置选择。

### 5.2 配置活动礼品流程

1. 运营进入“活动管理 > 活动详情 > 礼品配置”或“礼品中心 > 活动礼品配置”。
2. 选择活动。
3. 点击“添加礼品”。
4. 从礼品库弹窗搜索礼品 code、名称、品牌、类型。
5. 批量选择礼品后加入活动。
6. 设置排序、启停、活动内展示时间。
7. 如有特殊活动图或短标题，可做活动内覆盖。
8. 保存后消费者端活动页生效。

### 5.3 查看消费者端展示

建议在礼品库和活动礼品配置页增加“预览”能力：

1. 我的礼品卡片预览。
2. 活动页礼品卡片预览。
3. 礼品详情弹层预览。

这能显著降低运营传错图、文字过长、图片比例不适配的问题。

## 6. 权限与风控

| 权限点 | 建议 |
|---|---|
| 礼品查看 | 活动运营、客服、产品、测试可见 |
| 礼品同步/编辑 | 限制给礼品运营、管理员 |
| 礼品启停 | 建议单独权限，避免误下线 |
| 活动绑定礼品 | 活动运营可操作 |
| 活动内覆盖主图/名称 | 建议记录操作日志 |
| 批量启停/删除 | 高风险操作，需二次确认 |

删除建议改为“停用/归档”，不要物理删除。已被活动绑定或历史订单使用的礼品，不允许删除。

## 8. 研发改造建议

### 8.1 后端

1. 建立统一礼品主表，承接现有礼品管理和礼品展示管理的重复字段。
2. 建立活动礼品关系表，只保存活动维度的排序、状态、覆盖字段。
3. 前端接口统一返回礼品基础信息 + 活动关系信息。
4. 历史数据迁移时按礼品 code 合并，同 code 多条记录需要人工确认主数据。
5. 图片字段统一存储为资源 ID 或 CDN URL，并保留图片类型。

### 8.2 前端后台

1. 礼品库列表支持按 code、名称、品牌、类型、状态查询。
2. 礼品编辑页分为“基础信息、展示素材、详情信息、操作记录”Tab。
3. 活动礼品配置页只允许从礼品库选择礼品，减少手动输入。
4. 所有图片上传组件展示尺寸规范和实际使用场景。
5. 增加消费者端预览，降低发布后返工。

### 8.3 消费者端

1. 活动页礼品卡片读取活动礼品配置。
2. 我的礼品读取礼品主数据主图和名称。
3. 礼品详情读取详情图、视频、客服电话。
4. 活动内覆盖字段优先级高于礼品库默认字段。

## 9. 分阶段落地

### P0：配置收口

1. 在现有礼品管理中补齐图片类型：主图、活动展示图、详情图、物料图。
2. 活动配置礼品时改为从礼品库选择。
3. 去掉活动配置里重复录入礼品名称、code、客服电话的能力。

### P1：体验优化

1. 增加批量绑定礼品、批量启停、排序。
2. 增加消费者端预览。
3. 增加操作日志和图片上传校验。

### P2：治理与增长

1. 增加礼品素材复用、版本管理。
2. 增加礼品效果数据：曝光、点击、兑换。
3. 支持按礼品效果反向优化活动礼品选择。

## 10. 验收标准

1. 运营可在一个入口完成礼品基础信息、图片、详情、视频、客服电话维护。
2. 活动配置礼品时，只需要选择礼品并配置排序/启停，不再重复填写礼品基础资料。
3. 消费者端活动页、我的礼品、礼品详情展示的数据来源一致。
4. 同一礼品绑定多个活动时，默认展示一致；活动级覆盖只影响当前活动。
5. 已绑定活动或产生订单的礼品不可删除，只能停用或归档。
6. 图片上传前能看到尺寸、格式、大小要求，保存后能预览端侧效果。

## 11. 产品结论

本次改造不应简单理解为“把几个后台页面合并到一个页面”，而应该升级为“礼品主数据中心 + 活动礼品关系配置”。

推荐最终形态：

**礼品中心统一维护礼品资料，活动侧只做绑定和局部覆盖。**

这样既能满足业务想要的“一处维护”，又能保留活动运营的灵活性，避免图片、名称、详情、客服电话在多个后台入口重复配置导致的展示不一致和运营返工。
